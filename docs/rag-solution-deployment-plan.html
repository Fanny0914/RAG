文档三 · 基于 PDF 方案扩展

基于本地 Chroma 与云端大模型的 RAG 开发部署方案详解

本文基于仓库中 PDF《开发者实战：基于本地 Chroma 与云端通义千问搭建 RAG 系统》的核心思路， 对方案做进一步工程化展开。原文强调的是一条非常适合企业试点的路线：知识库和向量检索尽量本地化，大模型推理能力通过云端 API 获取。 这种“本地存知识，云端出智能”的组合，兼顾了隐私、成本和落地速度。

一、为什么这个方案适合企业试点？

很多企业在做 RAG 时，第一反应是“要不要先买 GPU、先建集群、先训练模型”。但对于多数知识问答场景来说，这样的起步方式过重了。 PDF 里的方案给了一个更现实的路径：把最敏感的知识库留在本地，把最昂贵的推理能力交给成熟云模型。

核心收益： 低成本启动、减少私有文档外流范围、部署快、调试直观、对中小企业和部门级项目尤其友好。

二、方案的核心架构

模块 建议实现 部署位置 作用
文档存储 企业本地文件夹 / 内网文件系统 本地或内网 保存原始制度、手册、FAQ、合同模板等资料
Embedding 模型 中文 embedding 模型，如 BGE-small-zh 本地设备 把文档片段和用户问题转成向量表示
向量数据库 Chroma 本地磁盘 保存向量索引，实现语义检索与持久化
编排框架 LangChain 等 本地服务层 串起加载、切片、检索、Prompt 组装和问答流程
大语言模型 云端模型 API，如通义千问 云端 在给定检索上下文的基础上生成最终回答

三、完整工作流：从导入知识到生成答案

  1. 文档收集：准备企业知识源，例如规章制度、产品手册、培训资料、售后脚本。
  2. 文档清洗：统一编码、去除噪音页、拆分过长文档、补充标题和来源元数据。
  3. 文本切片：把长文切成适中的 chunk。PDF 中示例为 chunk_size=600、chunk_overlap=100。
  4. 本地向量化：用本地 embedding 模型把每个 chunk 转为向量。
  5. 写入 Chroma：将向量和元数据存入本地持久化目录，例如 ./local_db。
  6. 用户提问：收到问题后，先把问题向量化。
  7. 相似度检索：在 Chroma 中召回最相关的 3~5 个片段。PDF 示例使用的是 k=3。
  8. Prompt 组装：把检索片段、来源信息与用户问题拼接成面向大模型的上下文。
  9. 云端生成：将精炼后的 Prompt 发给云端 LLM API 获取最终答案。
  10. 结果返回：在本地界面中显示答案，并可进一步展示引用片段、来源文件名和时间戳。

四、为什么“本地 Chroma + 云端模型”很实用？

  • 知识不必整体上传：原始文档和向量索引留在本地，更适合敏感知识场景。
  • 不需要高端显卡：复杂推理由云端 API 承担，本地设备压力较小。
  • 更新简单：新增或修改文档时，重新切片并更新向量库即可。
  • 易于试点：非常适合用普通办公电脑完成 POC，再视情况升级为服务化部署。

五、结合 PDF 示例，可落地的开发骨架

PDF 给出的参考栈包含 langchain-community、dashscope、chromadb、 sentence-transformers。其核心逻辑可以概括为下面四步： 

1. 读取本地文档目录
2. 按固定 chunk 规则切片
3. 用本地 embedding 模型构建并持久化 Chroma 向量库
4. 用云端大模型 API 通过 RetrievalQA 链完成问答

如果要把这个方案从“示例代码”推进到“企业内部可用”，建议增加以下工程化能力：

  • 增量更新任务：自动发现新文档，只重建受影响索引。
  • 来源追踪：每个回答展示命中文档与片段，便于信任建立和人工校验。
  • 权限隔离：不同部门调用不同知识空间，避免越权检索。
  • 日志与审计：记录问题、召回结果、模型回答、耗时与错误。
  • 回退机制：检索不足时返回“未找到足够依据”，而不是编造答案。

六、推荐的部署阶段

阶段 1：单机 POC

适合部门试点。知识文件、embedding、Chroma 和应用代码都在一台机器上，云端只负责 LLM 推理。 优点是部署快、定位问题容易；缺点是并发和权限治理能力有限。

阶段 2：内网服务化

把文档处理、检索服务、问答服务做成局域网内可调用的接口，前端接企业门户、IM、客服台或管理后台。 这一步最关键的是统一日志、权限与知识更新流程。

阶段 3：企业级运营

当场景扩展到多个部门时，可以按业务线拆分知识库，对索引更新、模型调用成本、答案质量、命中率和敏感内容进行长期运营。

七、企业上线时必须补齐的四类能力

能力 为什么重要 建议做法
知识治理 脏数据会直接导致错误召回 建立文档模板、版本号、更新时间与责任人
安全治理 企业最担心敏感信息泄露 API Key 保密、传输加密、敏感字段脱敏、权限隔离
质量治理 RAG 不等于天然准确 建立问答评测集，持续观察召回率与答案引用质量
成本治理 调用量上来后费用会快速增长 控制上下文长度、限制无效请求、按场景选择模型档位

八、一个更现实的企业建议

这个方案最适合做“高价值、低风险”的第一批场景，例如内部知识问答、售前资料助手、制度查询、标准作业指导查询。 它不追求一上来就全自动执行复杂任务，而是先让企业拥有一个可控、可信、可迭代的知识增强问答底座。

一句话总结： 先把企业私有知识稳稳地留在本地，再借助云端大模型把“理解与生成”能力接进来，这是很多团队做 RAG 的最优起跑姿势。
